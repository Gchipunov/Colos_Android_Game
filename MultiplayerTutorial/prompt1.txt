Building a networked 2D platformer using UDP and POSIX sockets requires a robust architecture to handle the "unreliable" nature of the protocol. Since UDP doesn't guarantee delivery or order, we'll implement a Client-Server model where the networking logic lives on a dedicated thread to prevent network "hiccups" from freezing your frame rate.

The Architecture
To keep the game smooth, we use a Thread-Safe Queue (or simple shared buffer with mutexes) to pass data between the Main Thread (Rendering/Logic) and the Network Thread (I/O).

1. Data Structures & Headers
First, we define our packet structure and the shared state.

C
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <pthread.h>
#include <unistd.h>
#include <arpa/inet.h>
#include <sys/socket.h>

// Simple packet structure
typedef struct {
    uint32_t packet_id;
    float x, y;
    int player_id;
} GamePacket;

// Shared data between threads
typedef struct {
    GamePacket latest_packet;
    pthread_mutex_t lock;
    int running;
} NetworkState;
2. The Network Thread (Receiver)
This thread loops indefinitely, waiting for incoming UDP packets. It updates the shared state whenever a new packet arrives.

C
void* network_thread_func(void* arg) {
    NetworkState* state = (NetworkState*)arg;
    int sockfd;
    struct sockaddr_in servaddr, cliaddr;

    // Create socket
    if ((sockfd = socket(AF_INET, SOCK_DGRAM, 0)) < 0) {
        perror("Socket creation failed");
        return NULL;
    }

    memset(&servaddr, 0, sizeof(servaddr));
    servaddr.sin_family = AF_INET;
    servaddr.sin_addr.s_addr = INADDR_ANY;
    servaddr.sin_port = htons(8080);

    // Bind socket
    if (bind(sockfd, (const struct sockaddr *)&servaddr, sizeof(servaddr)) < 0) {
        perror("Bind failed");
        return NULL;
    }

    GamePacket buffer;
    socklen_t len = sizeof(cliaddr);

    printf("Network thread started. Listening on port 8080...\n");

    while (state->running) {
        // recvfrom is blocking, which is fine on a separate thread
        ssize_t n = recvfrom(sockfd, &buffer, sizeof(GamePacket), 
                             MSG_WAITALL, (struct sockaddr *)&cliaddr, &len);
        
        if (n > 0) {
            pthread_mutex_lock(&state->lock);
            state->latest_packet = buffer; // Update shared state
            pthread_mutex_unlock(&state->lock);
        }
    }

    close(sockfd);
    return NULL;
}
3. The Main Thread (Game Loop)
The main thread reads from the shared state to update the position of entities on screen.

C
int main() {
    NetworkState state = {0};
    state.running = 1;
    pthread_mutex_init(&state.lock, NULL);

    pthread_t net_tid;

        fprintf(stderr, "Failed to create thread\n");
        return 1;
    }

    // Mock Game Loop

        pthread_mutex_lock(&state.lock);

Since you're building a platformer, standard UDP isn't enough to make it feel "good." You should look into these three concepts next:

What about network prediction and circular buffer ?

Gemini said
Network prediction and circular buffers are the "gold standard" for making a 2D platformer feel like a local game rather than a laggy mess. Since UDP is unreliable and has latency, these techniques hide the delay from the player.

1. The Circular Buffer (The Time Machine)
In networking, a circular buffer (or ring buffer) acts as a historical log of your game state.

Why use it? When the server sends a correction (e.g., "Actually, you were at X=50 at Time 100"), your client needs to look back at its own history to see what it was doing at Time 100 to figure out how much it needs to correct.

Implementation Concept:
Instead of one variable for player_x, you have an array of structs where the index is frame % BUFFER_SIZE.

C
#define STATE_HISTORY 120 // Store 2 seconds of history at 60fps

typedef struct {
    uint32_t frame_id;
    float x, y;
    float velocity_x, velocity_y;
    int input_mask; // Which buttons were held
} PlayerState;

PlayerState client_history[STATE_HISTORY];
2. Client-Side Prediction (The "Lie")
Without prediction, if you press "Right," your character doesn't move until the packet goes to the server and comes back (Round Trip Time). With 100ms ping, the game feels "heavy."

The Process:

Input: Player presses "Right."

Predict: The client immediately moves the player on screen and saves this state into the Circular Buffer.

Send: Send the input (not the position!) to the server.

Wait: Continue playing as if everything is fine.

3. Server Reconciliation (The "Correction")
Eventually, the server sends a packet back saying, "At Frame 100, you were actually at X=48" (perhaps because you bumped into an enemy the client didn't see yet).

How to handle it:

Check: Compare the Server's Frame 100 position with your client_history[100 % STATE_HISTORY].

Snap/Rewind: If the difference is large (the "Error Threshold"), you must Teleport the player to the server's version of Frame 100.

Re-play: This is the magic part. You then loop from Frame 101 up to the current frame and re-apply all the inputs stored in your circular buffer.

[!IMPORTANT]
To the player, this looks like a tiny "jitter" or "snap" rather than a 100ms delay on every single movement.

4. Integration with POSIX Sockets & Threads
Because you are using a separate Network Thread, your Shared State needs to include this buffer.

Network Thread: Receives the "Authoritative State" from the server and writes it into a "Incoming Server Packet" slot.

Main Thread: 1.  Processes local input.
2.  Performs Prediction (updates current position).
3.  Checks the "Incoming Server Packet" slot.
4.  If a new server packet exists, it performs the Rewind and Replay logic using the circular buffer.

Simplified "Replay" Logic in C:
C
void reconcile_player(PlayerState *history, PlayerState server_state) {
    int index = server_state.frame_id % STATE_HISTORY;
    
    // Check if the server disagrees with our history
    float error = distance(history[index].x, history[index].y, server_state.x, server_state.y);
    
    if (error > 0.1f) { // Error threshold
        // 1. Correct the past
        history[index] = server_state;
        
        // 2. Re-simulate from that point to the 'now'
        for (int i = server_state.frame_id + 1; i <= current_frame; i++) {
            int prev = (i - 1) % STATE_HISTORY;
            int curr = i % STATE_HISTORY;
            
            // Apply physics using the input we saved for that frame
            history[curr].x = history[prev].x + (history[curr].input_mask & KEY_RIGHT ? 5.0f : 0.0f);
            // ... apply rest of physics ...
        }
    }
}

Yes, absolutely. In fact, running a "light" version of the physics on the client is mandatory for modern 2D platformers to feel responsive. This is exactly how games like Celeste or Dead Cells handle networking.

The trick is to use a Deterministic Physics approach where the client and server run the exact same math, but the server has the "final say."

1. The "Light Logic" Split
You don't need to sync every single pixel. You only run the physics that affects the Local Player immediately.

Logic Type	Client Role (Predictive)	Server Role (Authoritative)
Player Movement	Runs every frame (Immediate feedback).	Runs every frame (Verification).
Gravity/Collisions	Simplified (AABB vs. Tilemap).	Full (All entities + triggers).
Projectiles	Visual only (Spawn immediately).	Damage calculation (Did it hit?).
Other Players	Interpolation (Smooth sliding).	Logic master (Where are they?).
2. Implementing the Circular Buffer with Logic
To make this work with your POSIX threads, your "Light Logic" needs to be able to rewind. Instead of just storing positions, you store the Inputs that caused those positions.

The Structure

typedef struct {
    float x, y;
    float vx, vy;

    uint32_t frame_id; 
} PlayerState;

// Our "Time Machine"
PlayerState client_history[HISTORY_SIZE];

3. The "Rewind & Replay" Loop
When the Network Thread receives a packet from the server saying, "At Frame 100, you were actually at X=10.5," the Main Thread does this:

Find the error: Compare client_history[100 % 128].x with the server's 10.5.

Snap to Truth: If the difference is > 0.5 pixels, set x = 10.5 for frame 100.

Re-simulate (The Light Logic): Loop from frame 101 to the current frame, re-applying gravity and velocity based on the Inputs already stored in the buffer.

C
void run_light_physics(PlayerState *state) {
    // Simple AABB / Euler Physics
    if (state->inputs & 1) state->vx = -5.0f; // Left
    if (state->inputs & 2) state->vx = 5.0f;  // Right
    
    state->vy += 0.5f; // Gravity
    state->x += state->vx;
    state->y += state->vy;

    // Basic Tilemap Collision (Lightweight)
    if (state->y > GROUND_Y) {
        state->y = GROUND_Y;
        state->vy = 0;
    }
}
4. Handling the Thread Synchronization
Since your Network Thread is separate, you must ensure the Main Thread doesn't "miss" a server correction while it's busy rendering.

Step 1: Network Thread receives UDP packet → Places it in a volatile shared "ServerUpdate" struct.

Step 2: Main Thread (Start of frame) → Locks Mutex, copies "ServerUpdate", unlocks.

Step 3: Main Thread → Performs the Rewind/Replay if the frame_id is new.

Why "Light" Logic is better than "Full" Logic:
Performance: You only simulate the local player's collisions 60–120 times during a "rewind," which takes microseconds.

Consistency: By keeping the physics code simple (using fixed-point math or floats carefully), you ensure the Client and Server arrive at the same result 99% of the time.

Lag Compensation: It allows the player to "feel" 0ms latency even if they have 150ms ping.

Important: Fixed Timestep
For this to work, your physics cannot be tied to your FPS. If one player runs at 144Hz and another at 60Hz, their "Frame 100" will represent different times.

Is it wise to take input for controls and text fields off a seperate thread ?

Gemini said
The short answer is no, it is generally not wise to handle input on a separate thread. In almost all modern operating systems and graphics APIs (like OpenGL with GLFW, SDL, or Win32), input must be processed on the Main Thread (Thread 0).

Here is why you should keep input on the main thread and how to handle it properly in a networked environment.

1. OS Constraints: The "Main Thread" Rule
Most Operating Systems (Windows, macOS, Linux/X11) tie the window's event queue to the thread that created the window.

If you try to call glfwPollEvents() or SDL_PollEvent() on a background thread, the program will often crash, hang, or simply fail to receive any data.

macOS is particularly strict: the "App Kit" UI loop must run on the very first thread created by the process.

2. The "Input Latency" Paradox
If you move input to a separate thread to "save time," you often actually increase latency.

Main Thread Input: You press "Jump" → The engine processes physics → The frame is drawn. (Minimal delay).

Separate Thread Input: You press "Jump" → Input Thread saves it to a mutex → Main Thread is busy rendering → Main Thread finally checks the mutex on the next frame. You've just added a frame of lag.

3. How to do it right (The Networked Way)
Instead of a separate thread, use a Fixed Timestep Accumulator. This allows your input and physics to stay perfectly synced for your "Light Logic" and "Circular Buffer," while your Rendering/OpenGL runs as fast as the monitor allows.

The "Input Sampling" Pattern
Poll (Main Thread): Capture the raw keyboard state (e.g., is_pressed[KEY_SPACE]).

Buffer: Store that input state into your Circular Buffer for the current frame_id.

Ship: The Network Thread then reads from this buffer to send the packet to the server.

C
// RUNNING ON MAIN THREAD
void game_loop() {
    while (running) {
        // 1. Get Input (Must be Main Thread)
        glfwPollEvents(); 
        uint8_t current_input = capture_input_mask();

        // 2. Fixed Timestep Accumulator
        while (accumulator >= dt) {
            // Store input in history for prediction/networking
            client_history[current_frame % 128].inputs = current_input;
            client_history[current_frame % 128].frame_id = current_frame;

            run_light_physics(&client_history[current_frame % 128]);
            
            current_frame++;
            accumulator -= dt;
        }

        // 3. Render (OpenGL)
        render_game(); 
    }
}
4. What about Text Fields?
Text fields are even more dangerous on separate threads because they involve UI State.

If the user types "HELLO" and your logic thread tries to read that string while the input thread is still writing the 'O', you get a Race Condition or a crash.

The Solution: Use a "Command Queue." The Main Thread captures the text input and, once the user hits "Enter," it copies the final string into a thread-safe buffer for the Network Thread to send.

UDP Socket I/O	Network	recvfrom is blocking; keeps network lag from freezing the screen.
Physics/Logic	Main	Must stay in lock-step with input for prediction to work.

1. The Bitmask Input Pattern
Instead of sending strings or large structs, we assign each key a bit.

C
// Define our bits
#define INPUT_LEFT  (1 << 0) // 00000001
#define INPUT_RIGHT (1 << 1) // 00000010
#define INPUT_JUMP  (1 << 2) // 00000100
#define INPUT_DASH  (1 << 3) // 00001000

// In your Main Loop (using GLFW as an example)
uint8_t capture_input_mask(GLFWwindow* window) {
    uint8_t mask = 0;
    if (glfwGetKey(window, GLFW_KEY_A) == GLFW_PRESS) mask |= INPUT_LEFT;
    if (glfwGetKey(window, GLFW_KEY_D) == GLFW_PRESS) mask |= INPUT_RIGHT;
    if (glfwGetKey(window, GLFW_KEY_SPACE) == GLFW_PRESS) mask |= INPUT_JUMP;
    if (glfwGetKey(window, GLFW_KEY_LEFT_SHIFT) == GLFW_PRESS) mask |= INPUT_DASH;
    return mask;
}
Why this is great: When your Network Thread sends the packet, it's just sending one byte. When the Server receives it, it checks the jump bit with: if (packet.inputs & INPUT_JUMP) { /* trigger jump */ }.

2. The Linux Equivalent of AttachThreadInput
The Windows AttachThreadInput function is used to share an input queue (keyboard/mouse focus) between two different threads.

On Linux (X11/Wayland), there is no direct 1-to-1 equivalent because the architecture is fundamentally different.

In Linux, input is handled by the X Server (or Wayland Compositor). The focus isn't "attached" to a thread; it's attached to a Window ID.

If you are trying to share input between threads:
You don't "attach" them; you Pipe them. Since the X11/Wayland event loop must stay on the main thread, you simply read the events there and push them into a thread-safe structure that other threads can read.

If you are trying to "Force Focus" or "Global Hook":
If your goal was to allow a background thread to "steal" or "inject" input (which is often why people use AttachThreadInput), here are the Linux alternatives:

Since you are using POSIX sockets, the most "Linux-way" to handle this is using a Thread-Safe Queue.

Main Thread: Calls XNextEvent (or glfwPollEvents).

Main Thread: Packs the result into that 1-byte bitmask.

Main Thread: Pushes that byte into a small thread-safe buffer.

Network Thread: Wakes up, grabs the byte, and calls sendto() on the UDP socket.

double get_time_ms() {
    struct timespec ts;
    clock_gettime(CLOCK_MONOTONIC, &ts);
    return (ts.tv_sec * 1000.0) + (ts.tv_nsec / 1000000.0);
}

typedef struct {
    uint8_t type; // PKT_TIME_SYNC
    double t1;    // Client send time
    double t2;    // Server receive time
} TimeSyncPacket;
The Network Thread Sync Logic:
C
void perform_sync(int sockfd, struct sockaddr_in* servaddr) {
    TimeSyncPacket pkt = { .type = PKT_TIME_SYNC, .t1 = get_time_ms() };
    
    sendto(sockfd, &pkt, sizeof(pkt), 0, (struct sockaddr*)servaddr, sizeof(*servaddr));
    
    TimeSyncPacket response;
    recvfrom(sockfd, &response, sizeof(response), 0, NULL, NULL);
    
    double t3 = get_time_ms();
    double rtt = t3 - response.t1;
    double latency = rtt / 2.0;
    
    // Offset = ServerTime - (ClientTime + Latency)
    double offset = response.t2 - (response.t1 + latency);
    
    printf("Sync Complete: RTT=%.2fms, Offset=%.2fms\n", rtt, offset);
}
3. Converting Time to "Frame IDs"
For your Circular Buffer and Prediction to work, both the client and server must agree that it is currently "Frame X."

Instead of raw milliseconds, we use the synced time to calculate a discrete frame number.

C
#define TICK_RATE 60.0 // 60 updates per second
#define MS_PER_TICK (1000.0 / TICK_RATE)

uint32_t get_server_frame(double offset) {
    double server_time = get_time_ms() + offset;
    return (uint32_t)(server_time / MS_PER_TICK);
}
4. Handling Jitter (Moving Average)
UDP latency spikes. If you calculate the offset once and never again, your clock will drift. If you calculate it every frame, your "Frame ID" will jump forward and backward randomly (jitter).

The Solution:

Perform a sync every 5–10 seconds.

Store the last 10 offset results in an array.

Use the Median or Average of those offsets to keep the clock stable.

Summary of the System
Main Thread: Runs at 60Hz (Fixed Timestep) using get_server_frame().

Network Thread: Periodically pings the server to update the offset variable (protected by a mutex).

Result: When you send an input packet, it's tagged with current_frame. When the server gets it, it knows exactly when that input happened relative to its own clock.

void run_server_loop(int sockfd) {
    struct sockaddr_in cliaddr;
    socklen_t len = sizeof(cliaddr);
    GamePacket pkt;

    while (running) {
        // 1. Wait for packet (with a 1ms timeout to keep the loop ticking)
        fd_set readfds;
        FD_ZERO(&readfds);
        FD_SET(sockfd, &readfds);
        struct timeval tv = {0, 1000}; 

        if (select(sockfd + 1, &readfds, NULL, NULL, &tv) > 0) {
            recvfrom(sockfd, &pkt, sizeof(pkt), 0, (struct sockaddr*)&cliaddr, &len);

            // If client says "Frame 500" but we are at "Frame 600",
            // the packet is too old. Discard or lag-compensate.

                continue;

            update_world_physics();
            broadcast_world_state(sockfd);
            next_tick_time += MS_PER_TICK;
            current_server_frame++;

    }

The server doesn't trust the client's position. It only trusts the client's Inputs.

4. Advanced: Anti-Cheat & Buffer Management
Since you are using UDP, packets will arrive out of order. The server should maintain a Jitter Buffer for each client.

Instead of processing a packet the microsecond it arrives, the server buffers about 2–3 frames worth of inputs.

This ensures that if Packet A arrives slightly after Packet B, the server can still execute them in the correct chronological order.

    float x, y;

typedef struct {
    GameState local_player;
    GameState remote_player;
    pthread_mutex_t lock;
    int is_running;
    int sockfd;

// --- Network Thread: The Receiver ---
void* network_thread_func(void* arg) {
    NetContext* ctx = (NetContext*)arg;
    GameState incoming;
    struct sockaddr_in from_addr;
    socklen_t addr_len = sizeof(from_addr);

    printf("[Net] Receiver thread started.\n");

    while (ctx->is_running) {
        // Blocking receive
        ssize_t n = recvfrom(ctx->sockfd, &incoming, sizeof(GameState), 0, 
                             (struct sockaddr*)&from_addr, &addr_len);
        
        if (n > 0) {
            pthread_mutex_lock(&ctx->lock);
            // Update the "Source of Truth" for the remote player
            ctx->remote_player = incoming; 
            pthread_mutex_unlock(&ctx->lock);
        }
    }
    return NULL;
}

// --- Main Thread: Logic & Rendering ---
int main() {
    NetContext ctx = {0};
    ctx.is_running = 1;
    pthread_mutex_init(&ctx.lock, NULL);

    // 1. Setup UDP Socket
    if ((ctx.sockfd = socket(AF_INET, SOCK_DGRAM, 0)) < 0) {
        perror("Socket creation failed");
        return 1;
    }

    memset(&ctx.server_addr, 0, sizeof(ctx.server_addr));
    ctx.server_addr.sin_family = AF_INET;
    ctx.server_addr.sin_port = htons(PORT);
    inet_pton(AF_INET, SERVER_IP, &ctx.server_addr.sin_addr);

    // 2. Start Network Thread
    pthread_t net_tid;
    pthread_create(&net_tid, NULL, network_thread_func, &ctx);

    printf("[Main] Game Loop starting...\n");

    // 3. Mock Game Loop (The "Main" Thread)
    uint32_t frame_counter = 0;
    while (frame_counter < 500) { // Run for ~8 seconds

        ctx.local_player.inputs = current_input;

Final Design Overview
Key Takeaways from this Code:
Separation of Concerns: The main thread handles simulation and sending, while the network thread handles the blocking recvfrom.

Minimal Locking: The mutex is only held for the duration of a struct copy. This prevents the network thread from stalling the renderer.

UDP Flow: Notice we don't connect(). Every sendto and recvfrom is an independent event, allowing the game to survive temporary packet loss without a "reconnect" screen.
